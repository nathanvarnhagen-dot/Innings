## 2026-07-22T21:28:02-07:00 — Interpret Firebase phone-auth 400 response

### User input

> I have the following output in my developer tools:
>
> Failed to initialize reCAPTCHA Enterprise config. Triggering the reCAPTCHA v2 verification.
> identitytoolkit.googleapis.com/v1/accounts:sendVerificationCode?key=AIzaSyAU2VzlnSOILn7CrzWIH-HMD3YmbMr0I6s:1  Failed to load resource: the server responded with a status of 400 ()

### Notable updates

- Confirmed from current Firebase documentation that the Enterprise message is a fallback to reCAPTCHA v2 and is not, by itself, the SMS-delivery failure.
- Confirmed that a failed `sendVerificationCode` request means Firebase rejected the verification request before an SMS was sent.
- Identified the highest-priority Firebase configuration checks: Phone sign-in provider enabled, SMS region policy permits the destination country, and the active local development host is authorized for the project.
- Noted that projects created after April 28, 2025 no longer include `localhost` as an authorized domain by default, and that new projects can default to denying all SMS regions.
- The request's Network-response body contains the decisive Firebase error message; the status code alone cannot distinguish an unauthorized domain from a disabled provider, region-policy rejection, invalid reCAPTCHA credential, or quota/throttling issue.

### Verification

- Reviewed the Firebase Phone Auth setup guide, Firebase FAQ, and Authentication limits documentation.
- Confirmed the application currently suppresses the Firebase error code, so the browser's failed request response is required for exact classification.

### Final output

The reCAPTCHA Enterprise line is not the root failure. Firebase is falling back to reCAPTCHA v2; the subsequent `400` from `sendVerificationCode` means Firebase rejected the SMS request, so no text was sent.

Check Firebase Console in this order:

1. **Authentication → Sign-in method:** enable the **Phone** provider.
2. **Authentication → Settings → SMS region policy:** explicitly allow the United States (new projects can default to denying every region).
3. **Authentication → Settings → Authorized domains:** add the exact development host you use, such as `localhost`; if you are browsing `127.0.0.1`, check that host as well. Firebase no longer adds `localhost` automatically for projects created after April 28, 2025.
4. Check the project’s SMS quota/rate limits and whether that number has been throttled.

Then, in DevTools, open the failed `sendVerificationCode` request and copy the **Response** body’s `error.message`. That exact value will identify the remaining cause (`OPERATION_NOT_ALLOWED`, `UNAUTHORIZED_DOMAIN`, `INVALID_APP_CREDENTIAL`, quota/throttling, and so on). The current UI hides that useful error, which is a code issue I can fix next.

Firebase’s current setup guide documents the phone-provider, SMS-region, domain, and reCAPTCHA requirements: [Firebase Phone Auth for web](https://firebase.google.com/docs/auth/web/phone-auth). The localhost policy change is documented in the [Firebase FAQ](https://firebase.google.com/support/faq), and SMS limits are in [Firebase Authentication Limits](https://firebase.google.com/docs/auth/limits).
